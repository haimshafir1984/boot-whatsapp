# מסמך תכנון: מסלול Twilio מתקדם

תאריך: 21.5.2026

מטרת המסמך היא להגדיר איך נרחיב את המערכת הקיימת כך שתתמוך בשני מסלולי WhatsApp:

1. `WEB_JS` - המסלול הקיים, דרך סריקת QR וחיבור WhatsApp Web.
2. `TWILIO_API` - מסלול מתקדם ורשמי דרך Twilio ו-WhatsApp Business Platform.

היישום בפועל יתחיל רק אחרי הקמפיין של סוף השבוע, כדי לא לסכן את הקמפיין הראשון שרץ עכשיו.

## תקציר שלבי הפיתוח

1. חיסכון במערכת הקיימת: הוספת תאריכי קמפיין, סטטוס קמפיין, וכיבוי WhatsApp/Chromium בין קמפיינים.
2. אבסטרקציה בלי Twilio: הפרדת לוגיקת הבוט מהחיבור הטכני ל-WhatsApp, בלי שינוי חיצוני ללקוחה.
3. מודל לקוחות וחיבור ברמת לקוחה: הוספת `connectionType` עם ברירת מחדל `WEB_JS`, והכנה למסלול `TWILIO_API`.
4. Twilio Provider בסיסי: קבלת הודעות דרך Webhook ושליחת הודעות טקסט דרך Twilio.
5. כפתורי תגובה מהירה: הוספת Quick Replies רשמיים במסלול Twilio.
6. מסך Admin פנימי לניהול Twilio: ניהול Subaccounts, מספרים, Webhooks וסטטוסי חיבור.

## שלב ראשון לפני Twilio: הורדת עלויות במסלול הקיים

לפני שמוסיפים Twilio, כדאי לבדוק שיפור פשוט יותר במערכת הנוכחית: להוריד עלויות על ידי כיבוי בין קמפיינים.

כיום, במסלול `WEB_JS`, העלות העיקרית מגיעה מזה שהשרת מחזיק את WhatsApp Web ואת Chromium/Puppeteer פתוחים כל הזמן. אם אין קמפיין פעיל, השרת עדיין ממשיך לשלם על זיכרון, גם אם אף אחד לא שולח הודעות.

הפתרון הבסיסי:

- בזמן קמפיין השרת והבוט פעילים.
- בין קמפיינים מכבים את ה-Deployment או את רכיב ה-WhatsApp/Chromium.
- לפני קמפיין חדש מדליקים מחדש ובודקים שה-WhatsApp מחובר.

הפתרון היפה יותר בהמשך:

- להוסיף בדף הניהול שדות של תאריך ושעת התחלה לקמפיין.
- להוסיף תאריך ושעת סיום לקמפיין.
- להוסיף סטטוס ברור: מתוזמן / פעיל / הסתיים.
- כשהקמפיין מתקרב, המערכת תדליק אוטומטית את רכיב ה-WhatsApp/Chromium.
- כשהקמפיין נגמר, המערכת תסגור את רכיב ה-WhatsApp/Chromium או תכבה את ה-Deployment.
- בדף הניהול תופיע בדיקת מוכנות: WhatsApp מחובר, Google מחובר, קמפיין פעיל, לינק תקין.

מבחינת חוויית לקוחה:

- הלקוחה תבחר מתי הקמפיין מתחיל ומתי הוא מסתיים.
- המערכת תציג לה האם הקמפיין מוכן לריצה.
- אם ה-WhatsApp לא מחובר לפני תחילת הקמפיין, המערכת תבקש חיבור מחדש.
- בזמן שהקמפיין לא פעיל, לא מובטח מענה אוטומטי לטריגרים.

אפשרות מתקדמת יותר:

- להשאיר את דף הניהול כשרת קל וזול.
- להפריד את הבוט לתהליך נפרד בשם `Bot Worker`.
- ה-`Bot Worker` נדלק רק בחלון זמן של קמפיין פעיל.
- כך הדשבורד יכול להישאר זמין, אבל Chromium לא צורך זיכרון 24/7.

זה לא מחליף את Twilio, אבל זה יכול לתת חיסכון מיידי במסלול הקיים ולהוריד משמעותית את העלות לקמפיינים קצרים של 24-48 שעות.

### שינוי בדף הניהול

בדף בניית/עריכת קמפיין יתווספו שדות חדשים:

- תאריך ושעת התחלה.
- תאריך ושעת סיום.
- סטטוס קמפיין: טיוטה / מתוזמן / פעיל / הסתיים.
- אפשרות הפעלה ידנית: "הפעל עכשיו".
- אפשרות סיום ידני: "סיים קמפיין".
- תצוגת מוכנות לפני קמפיין: WhatsApp מחובר, Google מחובר, קמפיין פעיל, לינק תקין.

בדף הקמפיינים הקיימים יוצג לכל קמפיין:

- מתי הוא מתחיל.
- מתי הוא מסתיים.
- האם הוא פעיל כרגע.
- האם הבוט מאזין כרגע.
- האם נדרשת פעולה מהלקוחה, למשל חיבור WhatsApp מחדש.

החוויה ללקוחה תהיה פשוטה:

- היא בונה קמפיין כמו היום.
- היא בוחרת מתי הקמפיין אמור לרוץ.
- המערכת מראה לה אם הכל מוכן.
- בזמן שהקמפיין לא פעיל, המערכת לא מבטיחה מענה אוטומטי להודעות טריגר.

### מה יקרה טכנית מאחורי הקלעים

בשלב הראשון לא חייבים לכבות את כל השרת. אפשר להתחיל בכיבוי החלק היקר: `whatsapp-web.js` ו-Chromium.

המשמעות:

- שרת Express ודף הניהול יכולים להמשיך לרוץ.
- תהליך WhatsApp/Chromium יופעל רק אם יש קמפיין פעיל או קמפיין שעומד להתחיל בקרוב.
- כשאין קמפיין פעיל, המערכת תסגור את לקוח WhatsApp ואת הדפדפן של Puppeteer.
- כך צריכת הזיכרון יורדת משמעותית, גם אם הדשבורד עדיין זמין.

מנגנון מוצע:

- להוסיף Scheduler פנימי שרץ כל דקה.
- הוא בודק האם יש קמפיין פעיל לפי תאריך התחלה/סיום.
- אם יש קמפיין פעיל והבוט כבוי, הוא מפעיל את WhatsApp Client.
- אם אין קמפיין פעיל והבוט דולק, הוא סוגר את WhatsApp Client.
- אפשר להוסיף חלון ביטחון, לדוגמה להדליק 10-15 דקות לפני תחילת הקמפיין.
- אם WhatsApp דורש QR מחדש, הדשבורד מציג ללקוחה שצריך להתחבר לפני תחילת הקמפיין.

כיבוי ברמת הקוד:

- קריאה ל-`client.destroy()` של `whatsapp-web.js`.
- וידוא ש-Puppeteer/Chromium נסגר.
- עדכון מצב פנימי: `botState.ready = false`, `botState.listening = false`.
- שמירת הסשן בדיסק כדי שבהפעלה הבאה לא יהיה צורך ב-QR אם החיבור עדיין תקף.

אפשרות חסכונית יותר, אך מורכבת יותר:

- להשתמש ב-Railway API או מנגנון חיצוני כדי לכבות/להדליק את ה-Deployment כולו.
- זה יכול לחסוך יותר, אבל אם השרת כולו כבוי, גם דף הניהול לא יהיה זמין.
- לכן לשלב ראשון עדיף לכבות רק את WhatsApp/Chromium, ולהשאיר את הדשבורד חי.

## עקרונות מרכזיים

- לא שוברים את המסלול הקיים. כל מה שעובד היום ב-`WEB_JS` חייב להמשיך לעבוד אותו דבר.
- בחירת המסלול היא ברמת לקוחה, לא ברמת קמפיין.
- מסלול Twilio מיועד ללקוחות מתקדמות בלבד, בתמחור שונה.
- אתה, כמנהל המערכת, מחליף ללקוחה מסלול. הלקוחה לא צריכה להתעסק עם הגדרות טכניות.
- אין צורך בשלב הראשון ב-Inbox או שמירת היסטוריית שיחות.
- אין צורך בשלב הראשון בכפתורים במסלול `WEB_JS`.
- דריסת שם איש קשר קיים נשארת ברירת מחדל.

## הארכיטקטורה הרצויה

המערכת תעבוד עם מושג חדש: `WhatsAppProvider`.

במקום שכל הלוגיקה תהיה תלויה ישירות ב-`whatsapp-web.js`, תהיה שכבה משותפת שאחראית על:

- קבלת הודעה נכנסת.
- בדיקה האם ההודעה תואמת טריגר של קמפיין פעיל.
- שאלת שם, אם האפשרות פעילה.
- יצירת שם איש קשר לפי הקמפיין.
- הכנסת שמירת איש הקשר לתור.
- שליחת הודעות אוטומטיות למשתמש.

ההבדל בין המסלולים יהיה רק בשכבת החיבור ל-WhatsApp:

| חלק במערכת | WEB_JS | TWILIO_API |
| --- | --- | --- |
| קבלת הודעות | מאזין של `whatsapp-web.js` | Webhook של Twilio |
| שליחת טקסט | `client.sendMessage` | Twilio Messages API |
| כפתורים | לא נתמך בשלב ראשון | Quick Replies רשמיים |
| חיבור לקוחה | QR / pairing code | מספר WhatsApp רשמי דרך Twilio |
| שימוש בטלפון הלקוחה | כן, אותו WhatsApp בטלפון | לא באותה אפליקציה, זה ערוץ API רשמי |

## מבנה חשבונות Twilio

ההמלצה היא לעבוד עם חשבון Twilio ראשי שלך ומתחתיו `Subaccount` לכל לקוחה מתקדמת.

המבנה:

```text
Twilio Account ראשי שלך
  ├─ Subaccount: לקוחה א
  │   └─ WhatsApp Sender / מספר של לקוחה א
  ├─ Subaccount: לקוחה ב
  │   └─ WhatsApp Sender / מספר של לקוחה ב
  └─ Subaccount: לקוחה ג
      └─ WhatsApp Sender / מספר של לקוחה ג
```

היתרונות:

- הפרדה בין לקוחות.
- קל יותר להבין שימוש ועלויות לכל לקוחה.
- אפשר לנתק או לשנות לקוחה אחת בלי לפגוע באחרות.
- לא צריך שכל לקוחה תנהל חשבון Twilio לבד.
- החוויה ללקוחה יכולה להיות פשוטה: מחובר / לא מחובר.

המשמעות העסקית:

- אתה אחראי יותר על ההקמה, החיוב והתמיכה.
- הלקוחה לא צריכה להבין Twilio.
- עדיין ייתכן שתידרש הגדרת Meta Business / WABA / מספר לכל לקוחה, לפי המסלול ש-Twilio מאפשר בפועל.

## מדיניות מספרי טלפון במסלול Twilio

ברירת המחדל במסלול `TWILIO_API` תהיה מספר WhatsApp חדש ונפרד לכל לקוחה מתקדמת.

לא ננסה לחבר את המספר האישי/הקיים של הלקוחה ל-Twilio כברירת מחדל, כי מספר שכבר פעיל באפליקציית WhatsApp או WhatsApp Business בטלפון לא יכול לשמש במקביל כערוץ WhatsApp Business Platform רגיל. כדי להעביר מספר קיים למסלול API, בדרך כלל צריך לנתק אותו מהאפליקציה הקיימת, וזה עלול לגרום לאובדן נגישות רגילה לשיחות הישנות ולבלגן תפעולי.

לכן ההמלצה העסקית והטכנית היא:

- לרכוש ללקוחה מספר וירטואלי חדש דרך Twilio או דרך מסלול מספרים מתאים.
- לחבר את המספר החדש ל-WABA / WhatsApp Sender של אותה לקוחה.
- ליצור את לינק הקמפיין למספר החדש.
- להשאיר את ה-WhatsApp הרגיל של הלקוחה בטלפון ללא שינוי.
- להציג זאת ללקוחה כהפרדה בין "הוואטסאפ הרגיל שלך" לבין "מספר הקמפיינים האוטומטי".

היתרונות:

- אין צורך למחוק את ה-WhatsApp הקיים של הלקוחה.
- אין פגיעה בהיסטוריית השיחות שלה.
- יש הפרדה ברורה בין שיחות רגילות לבין אוטומציית קמפיינים.
- קל יותר למדוד שימוש ועלויות לפי לקוחה.
- קל יותר לכבות/להחליף מספר בלי לפגוע בפעילות היומיומית שלה.

חיסרון שצריך לתווך נכון:

- המשתמשים יכתבו למספר חדש, לא בהכרח למספר שהכירו קודם. לכן סטטוסים, קישורי שיתוף וטקסטים בקמפיין צריכים להציג את המספר החדש בצורה טבעית וברורה.

## מה הלקוחה תראה בדף הניהול

ללקוחה רגילה במסלול `WEB_JS`:

- חיבור WhatsApp עם QR או קוד.
- חיבור אנשי קשר.
- בניית קמפיין.
- ניהול ניסוחים.
- קמפיינים קיימים.

ללקוחה מתקדמת במסלול `TWILIO_API`:

- לא תראה QR.
- תראה סטטוס חיבור: מחובר / בהקמה / לא מחובר.
- תוכל לבנות קמפיינים כמו היום.
- תוכל לנהל ניסוחים כמו היום.
- אם במסלול Twilio יש כפתורי תגובה, היא תוכל לבחור/להגדיר כפתורים מתוך ממשק פשוט.

חשוב: לא לחשוף ללקוחה הרגילה שדות כמו `Account SID`, `Auth Token`, Webhook URL או מספרי Twilio. אלה צריכים להיות במסך מנהל פנימי שלך.

## מה אתה תראה כמנהל מערכת

צריך להוסיף בעתיד מסך ניהול פנימי או אזור Admin שבו תוכל להגדיר לכל לקוחה:

- סוג חיבור: `WEB_JS` או `TWILIO_API`.
- שם לקוחה.
- מזהה לקוחה פנימי.
- Twilio Account SID של ה-Subaccount.
- Twilio Auth Token או API Key.
- מספר WhatsApp של Twilio.
- Webhook URL שצריך להגדיר בטוויליו.
- סטטוס חיבור.
- הערות הקמה.

בהמשך אפשר להוסיף גם:

- תצוגת שימוש לפי לקוחה.
- כמות הודעות שנשלחו.
- עלות משוערת.
- סטטוס אישור Meta/WABA.

## השפעה על משתמש הקצה

משתמש הקצה הוא האדם ששולח את הודעת הטריגר.

במסלול `WEB_JS`:

- הוא שולח הודעה למספר WhatsApp הרגיל של הלקוחה.
- הוא מקבל הודעות אוטומטיות כמו היום.
- הכל נראה כאילו הוא מדבר עם הלקוחה ב-WhatsApp הרגיל שלה.

במסלול `TWILIO_API`:

- הוא שולח הודעה למספר WhatsApp חדש שמחובר ל-Twilio עבור הקמפיינים.
- הוא יכול לקבל הודעות רשמיות עם כפתורים, אם נגדיר זאת.
- מבחינתו זו עדיין שיחת WhatsApp רגילה.
- המספר יכול להיות שונה מהמספר הרגיל של הלקוחה, ולכן צריך לשלב זאת נכון בקמפיין ובלינקים.

הנקודה החשובה: במסלול Twilio, ההודעות לא יופיעו בהכרח באפליקציית WhatsApp Business הרגילה של הלקוחה בטלפון. זה ערוץ API. לכן בשלב הראשון לא נבנה Inbox, כי המטרה היא קמפיינים אוטומטיים ולא ניהול שיחה ידנית.

## שלבי עבודה מומלצים

### שלב 0: הקפאה עד סוף הקמפיין

לא משנים קוד ריצה לפני סיום הקמפיין של סוף השבוע, אלא אם יש תקלה קריטית.

### שלב 1: אבסטרקציה בלי Twilio

מטרת השלב: להכין את הקוד לשני Providers, אבל להשאיר רק את המסלול הקיים פעיל.

פעולות:

- יצירת טיפוסים משותפים ל-WhatsApp Provider.
- חילוץ לוגיקת הטריגר והשיחה מתוך `src/whatsapp.ts` לשירות משותף.
- עטיפת `whatsapp-web.js` בתוך `WebJsProvider`.
- בדיקה שהמערכת עובדת בדיוק כמו היום.

תוצאה רצויה:

- אין שינוי חיצוני ללקוחה.
- אין שינוי למשתמש הקצה.
- הקוד מוכן להוספת Twilio בלי לערבב את הכל בקובץ אחד.

### שלב 2: מודל לקוחות וחיבור ברמת לקוחה

מטרת השלב: להוסיף למערכת הבנה של לקוחה וסוג החיבור שלה.

פעולות:

- הוספת שדה `connectionType`.
- ברירת מחדל לכל לקוחה קיימת: `WEB_JS`.
- שמירת הגדרות חיבור ברמת לקוחה, לא ברמת קמפיין.
- התאמת הדשבורד כך שיציג רק את האפשרויות הרלוונטיות למסלול של אותה לקוחה.

### שלב 3: Twilio Provider בסיסי

מטרת השלב: לקבל ולשלוח הודעות טקסט דרך Twilio.

פעולות:

- התקנת SDK של Twilio.
- יצירת `TwilioProvider`.
- יצירת endpoint ל-webhook נכנס.
- חיבור הודעות נכנסות לאותה לוגיקה עסקית קיימת.
- שליחת הודעות אוטומטיות דרך Twilio.

בשלב הזה עדיין לא חייבים כפתורים.

### שלב 4: כפתורי תגובה מהירה

מטרת השלב: להוסיף יכולת אינטראקטיבית רשמית ללקוחות Twilio.

פעולות:

- הגדרת מבנה לכפתורים בקמפיין או בהודעות ההמשך.
- שליחה דרך Twilio כ-Quick Replies.
- ב-`WEB_JS` להשאיר fallback פשוט לטקסט רגיל או להסתיר את האפשרות.

### שלב 5: מסך Admin פנימי לניהול Twilio

מטרת השלב: להפוך את ההקמה ליותר נוחה עבורך.

פעולות:

- מסך ניהול לקוחות.
- יצירת/שמירת פרטי Subaccount.
- הצגת Webhook URL.
- סטטוס חיבור.
- רשימת בדיקות לפני הפעלה ללקוחה.

## סיכונים שצריך לנהל

- שינוי מוקדם מדי בקוד הקיים עלול לפגוע בקמפיין פעיל.
- Twilio דורש תהליך חיבור עסקי שיכול לקחת זמן, במיוחד ללקוחות לא טכנולוגיות.
- אם כל לקוחה תפתח חשבון Twilio לבד, זה יאריך ויקשה על המכירה.
- אם הכל מנוהל אצלך, אתה מקבל יותר שליטה אבל גם יותר אחריות.
- Twilio API לא עובד כמו WhatsApp בטלפון. צריך לתכנן נכון את חוויית הלקוחה.

   - זול יותר.

   - עובד דרך מספר חדש/API של Twilio ולא דרך האפליקציה הרגילה בטלפון.
   - משאיר את המספר הקיים של הלקוחה מחוץ לסיכון.

## החלטות שכבר התקבלו

- Twilio יהיה ללקוחות מתקדמות בלבד.
- בחירת Provider תהיה ברמת לקוחה.
- לא נבנה Inbox בשלב הראשון.
- לא נשמור היסטוריית שיחות בשלב הראשון.

- לא מבצעים שינויי קוד לפני סיום הקמפיין, אלא אם יש תקלה קריטית.

## Runtime notes added 2026-06-26

### Multi-number Twilio routing

The admin Twilio webhook is designed to support more than one Twilio WhatsApp sender using the same incoming webhook URL. The routing decision should use the inbound `To` number and match it against the managed client's `twilioFrom`.

Required alignment for each Twilio client:

- Twilio Console incoming webhook URL points to the admin webhook with the configured token.
- Owner/client record has `twilioFrom=whatsapp:+...` for that client.
- Client Dokploy environment has `TWILIO_FROM=whatsapp:+...` for outbound replies.
- Client environment has `WHATSAPP_PROVIDER=TWILIO_API`.

### Flow contact-card step

`contact_card` can be used as a normal decision-flow step. It is independent of the old before/after-completion contact-card placement.

Runtime order:

1. Send the contact-card intro text.
2. Send the configured vCard/contact card.
3. Wait `max(BOT_REPLY_DELAY_MS, 4000)` milliseconds.
4. Continue to the next flow step.

If the contact-card step fails and a next step exists, the system waits 60 seconds and continues to avoid leaving the user stuck.

### Export and recovery actions

- Detailed XLS export splits campaign events into separate columns per event.
- Existing campaigns get the new XLS format automatically after deploy.
- `queue-unsaved` can queue all not-yet-saved campaign entrants at the end of a campaign without messaging them.

## Update - 2026-06-29 - Campaign Flow Builder UI

Step 3 of the Twilio campaign wizard was changed from a long mixed settings page into a timeline-based flow builder.

Design decisions:

- Step 3 should contain only conversation blocks that happen during the flow.
- Completion behavior and fallback behavior belong in Step 4.
- The default route for each block is the next timeline block, not immediate end.
- Explicit next-step selection is still available when a flow needs branching or jumping.

Current supported Step 3 block types:

- message
- question/buttons
- question/list
- score_question
- wait_reply
- media/file
- contact_card

Operational note:

Push does not change live clients unless Dokploy auto-deploy is active or a manual deploy is performed for that client.
