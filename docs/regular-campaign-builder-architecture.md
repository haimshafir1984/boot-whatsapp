# Regular Campaign Builder Architecture

מטרה: להפוך את ניהול הקמפיין הרגיל למסלול בנייה עצמית, כך שמנהל יוכל להוסיף שלבים ושדות נפוצים בלי שינוי קוד.

## עיקרון מרכזי

הקמפיין צריך להישמר כרשימת בלוקים אחידה, לא כאוסף שדות קשיחים במסך:

```ts
type CampaignBlock =
  | { type: 'send_message'; text: string }
  | { type: 'wait_for_signal'; text: string; timeoutMinutes: number; autoContinue: boolean }
  | { type: 'ask_name'; text: string; timeoutMinutes: number }
  | { type: 'save_contact' }
  | { type: 'send_links'; links: Array<{ label: string; url: string }> }
  | { type: 'send_files'; fileIds: string[] }
  | { type: 'question'; text: string; options: QuestionOption[] }
  | { type: 'human_handoff'; text: string; phone: string };
```

המסך הקיים צריך להישאר תואם לאחור, אבל מאחורי הקלעים להפוך את ההגדרות הקיימות לבלוקים.

## מסלול מומלץ

1. טריגר.
2. בלוק הודעת "שמרתי?" מסוג `wait_for_signal`.
3. בלוק שאלת שם מסוג `ask_name`.
4. בלוק שמירה כאיש קשר מסוג `save_contact`.
5. בלוק הודעת תודה מסוג `send_message`.
6. בלוק לינקים מסוג `send_links`.
7. בלוק קבצים מסוג `send_files`.
8. בלוקים נוספים לפי צורך: שאלת בחירה, קובץ לפי תשובה, מעבר לאנושי.

## מסך ניהול

המסך צריך לכלול:

- רשימת שלבים בסדר ריצה, עם גרירה למעלה/למטה.
- כפתור "הוסף שלב".
- סוגי שלבים מוכנים: הודעה, המתנה לתשובה, שאלת שם, שמירה כאיש קשר, לינקים, קבצים, שאלה עם אפשרויות, מעבר לאנושי.
- בדיקת קמפיין אליי שמריצה את כל הבלוקים בפועל מול מספר בדיקה.
- תצוגת סיכום לפני הפעלה: אילו הודעות, אילו קבצים, כמה זמני המתנה, ומה קורה אם אין תשובה.

## כללי אמינות

- כל שליחת הודעה חייבת retry קצר.
- כל שליחת קובץ חייבת retry ותיעוד כישלון.
- כישלון של בלוק אחד לא צריך לעצור בלוקים לא תלויים, למשל לינק שנכשל לא עוצר קובץ.
- בלוק `wait_for_signal` לא צריך להבין טקסט ספציפי. כל תשובה, ריאקציה או מדיה נחשבים כהתקדמות.
- auto-continue הוא fallback בלבד. אם הוא קורה הרבה, זה סימן לחוסר יציבות חיבור או חיכוך גבוה מדי בטקסט.

## שלב ביצוע הבא

השלב הבא המומלץ הוא להוציא מתוך `public/index.html` את אזור עריכת הקמפיין ל-Flow Builder בסיסי:

- להשתמש בהגדרות הקיימות כ-data source.
- לא לשנות עדיין את פורמט הריצה בקמפיינים קיימים.
- להוסיף UI שמייצר את אותם שדות קיימים דרך בלוקים.
- לאחר שהמסך יציב, להעביר גם את מנוע הריצה לקריאת `flowBlocks` מלא.
